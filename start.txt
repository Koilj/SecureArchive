3. Start local IPFS (Kubo) replicas — REAL daemons, not mocks

Kubo is free, self-hosted and fully offline. `scripts/ipfs_dev.sh` manages
two isolated daemons under ~/.securedata-ipfs (one-off init, then start).

cd /home/ruslan/working
./scripts/ipfs_dev.sh start     # launches node-a (API 5011) + node-b (API 5012)
./scripts/ipfs_dev.sh status    # show pid / pins / repo stats
./scripts/ipfs_dev.sh env       # print export lines for the backend

The script:
  - creates one Kubo repo per node with --profile=server --empty-repo
  - binds API, gateway and swarm to distinct localhost ports
  - removes public bootstrap peers (private local cluster)
  - peers node-a and node-b to each other so bitswap works offline
  - enables GC + StorageMax=5GB per node (pins are protected from GC)

Do NOT use mock_ipfs_node.py for production or manual demos. It is a
unit-test-only stub: no libp2p, no DHT, no real UnixFS DAG. Real tests
against Kubo live in tests/test_ipfs_real.py.

8. Bootstrap SecurityService

- Open the UI and click the bootstrap action.
- The browser activates the device with a WebAuthn passkey.
- The browser also creates three device-local keys:
  - data encryption ECDH key for modern key envelopes
  - Fabric signing key
  - WebAuthn/passkey credential
- The backend only issues a bootstrap invite/enrollment ticket and proxies Fabric CA enrollment.
- The backend does not keep the user private key or a server-side Fabric wallet.


9. Create users

- Sign in to the UI as `SecurityService` with passkey/WebAuthn.
- Open `Users` and issue invite tickets for new users.
- A new user opens the same UI, pastes the invite token, activates their device, and then signs in later with passkey/WebAuthn.


10. Current security model

- no password auth for users
- no `securedata_auth.db`
- no server-side user Fabric wallet directory
- public WebAuthn credential metadata is stored in the ledger user profile
- backend `/fabric/eval` and `/fabric/submit` use offline proposal/transaction signing
- the device signs Fabric proposal/transaction bytes locally and the backend only forwards signed payloads


11. When to use the recovery bundle

- Use it on a new browser or a new device to restore the local content-decryption key and local Fabric signing key.
- Use it after browser storage reset, browser profile cleanup, OS reinstall, or device replacement.
- Use it when the UI reports that the local private key or local Fabric key is missing on this device.
- Do not use it for normal daily sign-in on the same activated browser/device.
- The bundle is separate from passkeys: passkeys handle login, the recovery bundle restores device-local decryption and Fabric signing material.
